## What this is

**reqost** is a high-performance desktop API client (a Postman-collection viewer) built with [Wails v3](https://v3.wails.io/) — a Go backend with a Vue 3 + TypeScript frontend rendered in a native webview. Its defining design goal is handling very large Postman collections (50k+ items) smoothly. It does this by parsing `collection.json` into a SQLite index once, then serving the UI lazily from that index — heavy request content is never loaded during tree rendering.

## Commands

All builds go through [Task](https://taskfile.dev) (`Taskfile.yml`), which wraps the `wails3` CLI. The frontend uses `npm` by default (override with `PACKAGE_MANAGER=bun|pnpm|yarn`).

- `wails3 dev` — run with hot-reload (frontend + Go). Or `task dev`.
- `task build` — production build into `bin/` (delegates to `{OS}:build`).
- `task run` — build and run the desktop app.
- `wails3 generate bindings` — regenerate the TS bindings in `frontend/bindings/` after changing any Go service method signature. **Required** whenever a `CollectionService` method changes; the frontend imports these generated stubs directly.
- `go build ./...` — compile the Go side only (fast sanity check). The `build/ios` "function main is undeclared" error is expected on non-iOS hosts (platform build tags) — ignore it.
- `go test ./...` — run Go tests (`internal/httpclient` via `httptest`, `internal/script` via goja — no network needed).
- `cd frontend && npm run build` — type-check (`vue-tsc`) + Vite build (production). `npm run build:dev` skips minification.
- `go vet ./...` is the closest available lint.

### Environment gotchas (this machine)

- **`wails3` CLI is not installed.** The Taskfile exports a `WAILS_CLI` var that runs it via `go run` pinned to the alpha tag, and `task bindings` / `task dev` use it. If you call wails3 directly, use `go run github.com/wailsapp/wails/v3/cmd/wails3@v3.0.0-alpha2.103 generate bindings -clean=true`.
- **Production frontend build needs a native CSS-minifier binding** that npm fails to resolve. Use `task frontend:build` — it runs `task fix:lightningcss` first to create the missing symlink (it auto-detects the right arch suffix). Manual fallback: `ln -sf ../lightningcss-darwin-arm64/lightningcss.darwin-arm64.node frontend/node_modules/lightningcss/lightningcss.darwin-arm64.node` (arch suffix varies by platform).

### Server / Docker mode

The project supports a GUI-less HTTP server build via the `server` build tag:
- `task build:server` / `task run:server` — build/run `bin/reqost-server` (no native GUI deps).
- `task build:docker` / `task run:docker` — distroless image; container always listens on **8080**.

### Seeding test data

`scripts/seed/main.go` generates a large synthetic Postman collection for performance testing:
- `go run ./scripts/seed [outPath]` — defaults to `collection-50k.json`.

## Architecture

### Backend (Go)

The Go side exposes Wails **services** whose methods become callable from JS via generated bindings. `main.go` registers them and wires the webview window. There are five services, kept separate on purpose:
- `CollectionService` (`service.go`) — the collection index (import, tree, search, detail) **plus writes**: `SaveRequest`, `CreateRequest`/`CreateFolder`, `RenameNode`, `DeleteNode`, `MoveNode` (drag-drop reorder/reparent), `ExportCollection`. The SQLite index is the **source of truth** for edits. `ImportCollection` is **incremental**: shared ids keep their `detail` row (user edits survive), missing ids are deleted, new ids are inserted. The file watcher no longer auto-reimports.
- `ExecService` (`exec_service.go`) — request execution. `SendRequest(reqId, req, preScript, postScript)` runs the optional pre-request script (may mutate the request + variables), executes, then runs the test script, returning a `SendResult` (response + test results + console logs + the post-script variable map + any non-fatal script error). It injects a `Sender` (`makeSender`) so `pm.sendRequest` reuses the same client/cookie jar. Also exposes `GetCookies(url)`/`ClearCookies` over the session jar. Holds an `inFlight` map of `reqId → context.CancelFunc`; `Cancel(reqId)` aborts.
- `EnvService` (`env_service.go`) — load/save the environments document (thin wrapper over `internal/envstore`).
- `WSService` (`ws_service.go`) — live WebSocket connections (`coder/websocket`). `Connect`/`Send`/`Close` keyed by a frontend `connId`; incoming frames + state changes are pushed as `ws:event` Wails events (it has its own `SetEmitter`, wired in `main.go`).
- `GRPCService` (`grpc_service.go`) — dynamic unary gRPC via **server reflection** (`jhump/protoreflect` + `grpc`), no `.proto` files needed (grpcurl-style). `ListMethods(target)` enumerates services; `Call(target, "pkg.Service/Method", reqJSON)` invokes. The target scheme picks transport: `grpcs://` → TLS, `grpc://`/bare → plaintext (`parseTarget`).

Six internal packages do the real work:

- **`internal/collection`** — Postman Collection Format v2.1 parsing. `parser.go::ParseFile` reads the JSON and `flatten`s the nested folder/request tree into a flat `[]FlatItem` slice (each with a `ParentID`). It converts Postman body modes (`raw`/`urlencoded`/`formdata`/`graphql` → `BodyType` + `FormFields`/`GraphqlVars`) and request-level `auth` (bearer/basic/apikey → the frontend Auth JSON shape) so imported requests arrive fully typed. `types.go` holds the Postman structs; note `PostmanURL` has a custom `UnmarshalJSON` because Postman's `url` field is sometimes a string and sometimes an object. (`parser_test.go` covers the body/auth conversions.)
- **`internal/index`** — the SQLite index (driver: `modernc.org/sqlite`, pure-Go, no cgo). `db.go` owns the schema and `ImportItems`; `queries.go` owns the read paths. **The schema split is the core performance pattern:**
  - `tree` — lightweight node rows (id, name, parent_id, type, method, sort_order) for fast lazy tree rendering. Children are fetched on demand per folder.
  - `detail` — heavy per-request content (url, headers, body, scripts), read **only** via `GetRequestDetail` when a request is actually opened.
  - `search_fts` — an FTS5 virtual table for full-text search over name/url/body.
  - The DB lives in the user cache dir (`os.UserCacheDir()/reqost/index.db`), runs in WAL mode (`busy_timeout=5000`), and uses a single connection (`SetMaxOpenConns(1)`). `OpenAt(path)` exists for isolated tests.
  - **Gotcha:** do NOT mix `tx.Query` and `tx.Exec` inside one transaction — on modernc.org/sqlite + WAL it can self-lock with `SQLITE_BUSY`. `DeleteNode` reads the subtree ids (recursive CTE) on `db.conn` *before* opening a write-only transaction for this reason. (`mutate_test.go` guards the read-then-delete path.)
  - `mutate.go` holds the write paths (Save/Create/Rename/Delete/**Move**), each in a transaction that also keeps the FTS row in sync (`reindexFTS`). `MoveNode(id, newParentID, newIndex)` reads sibling ids on `db.conn` first (WAL precaution), rejects self-into-descendant cycles via `ErrInvalidMove`, then densely rewrites sibling `sort_order` in a single tx. `export.go` rebuilds the nested Postman v2.1 JSON from the flat rows using export-only structs (so the JSON tags are spec-clean — the import-side `collection.*` types are not reused for output).
  - The `detail` table also persists per-request UI state edited in the app: `body_type`, `form_fields`, `graphql_vars`, `grpc_method`, `auth_json`, `settings_json` (per-request execution overrides: timeout/follow-redirects/max-redirects/verify-ssl). These were added after the initial schema, so `db.go::migrate` runs idempotent `ALTER TABLE ADD COLUMN`s (duplicate-column errors ignored) on existing DBs. `SaveRequest`/`GetRequestDetail`/`toDetail` round-trip them so a GraphQL/form-data/gRPC/auth request keeps its shape across reloads. The Postman parser fills body modes (`raw`/`urlencoded`/`formdata`/`graphql`) and auth on import — `grpc_method` and `settings_json` are populated only by in-app edits.
  - **Incremental import:** `ImportItems` reads the existing `(id, name, url, body)` set on `db.conn` first, then in one transaction: deletes ids that the new parse no longer has, updates `tree` for ids shared with the import (preserving the `detail` row so user edits survive a re-import), inserts brand-new ids fully (tree + detail + FTS). FTS is only refreshed when a shared id's *name* changed, using the kept `detail.url`/`body` so search results stay consistent with the displayed data. `mutate_test.go::TestImportItemsPreservesEdits` locks this in.
- **`internal/watcher`** — wraps `fsnotify` to watch the imported collection file and trigger a debounced (500ms) re-import on change.
- **`internal/httpclient`** — the request execution engine. `Execute(ctx, Request) (*Response, error)` uses `net/http` + `httptrace` for a per-phase timing breakdown (DNS/connect/TLS/TTFB/total), caps buffered bodies at 50 MiB, and pools connections via a shared cookie jar + two cached transports (secure / `InsecureSkipVerify=true`). `Request` carries `protocol` (only `"http"` today), a `variables` map, an optional `auth`, and per-execution options whose **zero values keep safe defaults**: `TimeoutMs` (0=no deadline), `DisableRedirect` (false=follow), `MaxRedirects` (0=stdlib default 10), `InsecureSkipVerify` (false=verify TLS). Each `Execute` builds a thin `http.Client` around the chosen transport so per-call `CheckRedirect` + `context.WithTimeout` work without losing pooling. `interpolate.go` substitutes `{{key}}` placeholders (in URL, header keys/values, body, auth, and form fields) from the variables map before sending; unknown keys are left intact. Auth (`bearer`/`basic`/`apikey`) is applied as a header. `body.go::buildBody` handles `bodyType` = none/raw/json/urlencoded/formdata — urlencoded and multipart form-data are built from `FormFields` (a `file`-typed field streams the file at its path). `Header` is an ordered, toggleable key/value pair shared by request and response. Being a native client, it is **not subject to browser CORS**.
- **`internal/openapi`** — converts an OpenAPI 3 / Swagger 2 spec (JSON or YAML, parsed with `yaml.v3` which also reads JSON) into `[]collection.FlatItem`: operations grouped into folders by first tag under a root folder named after the spec title. Builds the URL from `servers[0]`/`host+basePath`, header/query params, and a JSON body synthesized from the `requestBody` example or schema (with `$ref` → `components/schemas`/`definitions` resolution, depth-capped). `CollectionService.PickImportOpenAPI` merges it via `index.AddItems` (insert without clearing — unlike `ImportCollection` which replaces). (`openapi_test.go` covers the conversion.)
- **`internal/script`** — runs Postman-style pre-request and test scripts in a `goja` (pure-Go JS) sandbox. `prelude.go` defines the `pm.*` API in JS on top of Go-injected hooks (`__host`/`__request`/`__response`/`__info`): environment/variables/globals/collectionVariables get/set/unset/toObject + `replaceIn`, `pm.test`, a chai-style `pm.expect` (equal/eql/include/match/lengthOf/oneOf/keys/a/above/below/property/satisfy/`deep.equal`/ok/true/false/empty + `.not`), `pm.response` (`.json`/`.text`/`.code`/`.headers.get` + `.to.have.status[Code]/header/jsonBody/body`), `pm.cookies` (stub), `pm.iterationData` (stub), `pm.info` (eventName/requestName/requestId), `console`, `btoa`/`atob`, legacy `tests{}`/`responseBody`/`responseCode`, and **`pm.sendRequest`** (backed by a `Sender` the caller injects). `RunPre`/`RunTests` take that `Sender` and an `Info` struct so scripts see request metadata. A 2s watchdog interrupts runaway scripts. Still a pragmatic **subset**, not 100% parity. `script_test.go` covers the common cases.
- **`internal/envstore`** — dumb JSON persistence for environments (named bags of `{key,value,enabled}` vars + an `activeId`), written atomically to `<cache>/reqost/environments.json`. The frontend owns all editing and resolves the active variable map itself; `ActiveVars()` exists server-side but the frontend currently resolves locally and passes `variables` into `SendRequest`.

**Import flow & events:** `ImportCollection` is fire-and-forget — it spawns `reimport` in a goroutine and returns immediately. `reimport` skips re-parsing if the file mtime matches the stored mtime, otherwise calls `ImportItems` which now **merges incrementally** (see `internal/index` notes above): existing detail rows for shared ids are preserved, so a re-import won't clobber the user's in-app edits. Progress is reported asynchronously by emitting Wails events the frontend listens for: `collection:importing`, `collection:ready`, `collection:error`.

### Frontend (Vue 3 + TS)

The UI is a Postman-style shell: an icon rail, a collection sidebar, a request-tab bar, and a workbench (request editor + response).
- `App.vue` — composes `IconRail` + (`Sidebar` | `HistoryPanel` | `SettingsPanel`) + `main`(`TabBar` + `RequestWorkbench`). Holds only the active left-rail `mode`. `Environments` opens as a modal rather than a panel.
- `IconRail.vue` — leftmost vertical rail switching modes (Collections / Environments / History / Settings — all wired) plus the light/dark theme toggle.
- `composables/useSettings.ts` + `SettingsPanel.vue` — app-wide defaults persisted to `localStorage` (default `timeoutMs`, follow-redirects, max-redirects, verify-ssl, theme via `useTheme`). The Settings sidebar mode surfaces them; per-request overrides set on the workbench's Settings subtab take precedence at send time.
- `composables/useTree.ts` — owns the tree state as a **module-level** `shallowRef<FlatNode[]>` (one global flat list, not per-component). Expand/collapse splices children into/out of this flat array; this flat-list + virtual-scrolling design is what keeps 50k-node trees responsive. Nodes are `markRaw` to skip Vue reactivity overhead.
- `composables/useTabs.ts` — **module-level** reactive store of open request tabs. Each `ReqTab` holds its own editable request (name/method/url/params/headers/body/auth/scripts/description) + last response, so switching tabs preserves edits/results. `openRequest(node)` lazy-loads detail via `GetRequestDetail`; `openAdhoc(req)` opens a non-collection request (used by History). Unsaved-edit detection is snapshot-based: `markClean(tab)` baselines `tab.clean` at load/save, and `isDirty(tab)` compares the live `snapshot()`; `TabBar` shows a dot when dirty. The request **name** (editable in the workbench header) and **description** (Settings sub-tab) round-trip through Save.
- `composables/useEnv.ts` — module-level environment store (loads/saves via `EnvService`, debounced). `activeVars` resolves the active environment's enabled vars to a `Record<string,string>` that the workbench passes into `SendRequest` for `{{var}}` interpolation. Also owns the environments-modal open state.
- `composables/useHistory.ts` — module-level request history persisted to `localStorage` (cap 200). The workbench calls `record()` after each send; `HistoryPanel` lists entries and re-opens them via `openAdhoc`.
- `composables/url.ts` — `parseQuery`/`buildUrl`/`baseOf` keep the Params editor and the URL field in sync. Deliberately string-based (not `URLSearchParams`) so `{{variables}}` and key order/disabled state survive.
- `EnvironmentsModal.vue` — environment CRUD + variable editor (opened from the rail's Environments icon or the tab-bar gear). `HistoryPanel.vue` — the History rail mode.
- **Vue gotcha:** literal `{{ }}` can't appear inside template interpolation (the tokenizer mis-parses `}}` inside a string). Put such strings in a script const and bind them (see `URL_PLACEHOLDER`/`VAR_HINT` in `RequestWorkbench.vue`).
- `composables/useTheme.ts` — module-level theme ref; toggling sets `data-theme` on `<html>` and persists to `localStorage`. All component CSS reads tokens from `public/style.css` (`--bg`, `--text`, `--accent`, …) so the switch is instant.
- `Sidebar.vue` — `vue-virtual-scroller` `RecycleScroller`, debounced search (200ms), `collection:*` event listeners that reload the tree after import. Clicking a request calls `useTabs().openRequest`. Right-click opens a context menu (New Request/Folder, Rename, Delete, Copy as cURL); the header `+` menu does New/Import/Export. Import/Export use **native file dialogs** (`PickImport`/`PickExport` → Wails `app.Dialog`), and create/rename/delete use the in-app `DialogModal` (`useDialog.ts`) — **not** `window.prompt`/`confirm`, which silently return null in macOS WKWebView and would dead-end every action. Tree mutations update the flat list in place via `useTree`'s `refreshNode`/`removeNode`/`reloadChildren` (no full reload, so expansion is preserved). **Drag-and-drop** reorders/reparents nodes: each row is `draggable`, hover splits into before/after/into (middle 33–66% of a folder = "into") with a CSS indicator, drop calls `MoveNode(id, newParentId, newIndex)` then refreshes the affected parents. Disabled while a search filter is active.
- `TabBar.vue` — the open-request tabs (method color, middle-click / ✕ to close).
- `RequestWorkbench.vue` — binds to the active tab: URL bar (method/URL/**Send**↔**Cancel**/**Save** → `SendRequest`/`Cancel`/`SaveRequest`) + request sub-tabs (Params/Auth/Headers/Body/Pre-req/Tests/**Settings**) + response panel (Body/Headers/Cookies/**Test Results** + console logs). The Settings subtab exposes per-request execution overrides (timeout / follow redirects / max redirects / verify SSL — each a tri-state Inherit/On/Off select against the app default) plus a Description editor. `send()` resolves overrides into the explicit `timeoutMs`/`disableRedirect`/`maxRedirects`/`insecureSkipVerify` fields the backend `Request` expects. Pre-req/Tests are JS script editors run by `internal/script`; their results show in Test Results, and `pm.environment.set` changes are merged back into the active environment via `useEnv.applyVars`. Body supports none/raw/json/urlencoded/form-data (file fields take a path). `toDetail` (in `useTabs`) serializes a tab back to the index `RequestDetail` for Save — including the per-request `settings` JSON.
- `composables/curl.ts` — `toCurl` builds a "Copy as cURL" string (values emitted verbatim, including `{{vars}}`).
- **Multi-protocol routing:** `RequestWorkbench` picks a UI by URL scheme — `ws://`/`wss://` → `WsConsole.vue` (connect/send/message log, backed by `useWs.ts` listening to `ws:event`), `grpc://`/`grpcs://` → `GrpcConsole.vue` (target + reflect-to-list-methods + JSON request/response), else the normal HTTP workbench. New WebSocket/gRPC tabs are created from the sidebar `+` menu via `openAdhoc`. These consoles build their requests on the frontend, so they resolve `{{vars}}` via `composables/interpolate.ts` (HTTP requests interpolate in Go instead). The response **Cookies** tab reads the session jar via `GetCookies(url)`.
- **GraphQL** is a Body type: a query + variables editor that `send()` packs into a JSON `{query, variables}` POST body. `body_type`/`form_fields`/`graphql_vars`/`grpc_method`/`auth_json`/`settings_json` all persist on `detail`, so a GraphQL/form-data/gRPC/auth request keeps its full shape across reloads.
- **Collection Runner** — `useRunner.ts` runs every request under a node (`ListRequestsUnder`) sequentially, threading the variable map across requests (so `pm.environment.set` carries forward) and collecting per-request status + test pass counts; `RunnerModal.vue` shows progress. Launched from the sidebar folder menu ("Run Folder") or header ("Run Collection").
- `public/style.css` — global reset + theme tokens (dark default, `[data-theme="light"]` overrides). It was stripped of the Wails template's centered-page / `user-select:none` boilerplate; form fields and response views re-enable text selection via `.selectable`.
- `frontend/bindings/reqost/` — **generated**, do not hand-edit. Regenerate with `task bindings` (it shells out to the pinned `go run github.com/wailsapp/wails/v3/cmd/wails3@…`).

### CI / release / auto-update

- **`.github/workflows/build.yml`** — on push to `master`, PRs, and tags `v*`. Builds natively on `macos-14` (darwin/arm64), `macos-13` (darwin/amd64), `ubuntu-latest` (linux/amd64, needs `libwebkit2gtk-4.1-dev`), `windows-latest` (windows/amd64). Installs the `wails3` CLI via `go install …@v3.0.0-alpha2.103`, runs `wails3 generate bindings` + `npm run build`, then a direct `go build -trimpath -ldflags "-s -w -X reqost/internal/update.Version=… -X reqost/internal/update.RepoSlug=…"` so the Taskfile build path is bypassed in CI (no Docker cross-compile dance). Packages as `reqost-{os}-{arch}.tar.gz` (unix) / `.zip` (windows) with a sibling `.sha256`. On a tag push, a `release` job downloads all artifacts and ships them via `softprops/action-gh-release@v2` with auto-generated notes.
- **`internal/update`** + **`update_service.go`** — manual self-update. `update.Check()` hits `api.github.com/repos/{RepoSlug}/releases/latest`, picks the `reqost-{GOOS}-{GOARCH}.{tar.gz|zip}` asset, compares the tag to the linker-injected `Version`. `update.Apply()` downloads + verifies the `.sha256` + extracts the binary + calls `minio/selfupdate` (cross-platform, including Windows in-place replace). Exposed to JS as `UpdateService.CurrentVersion/RepoSlug/CheckForUpdate/ApplyUpdate`. `SettingsPanel.vue` has the user-facing Check/Install UI; install success prompts a manual quit/relaunch.
- `Version` and `RepoSlug` default to `"dev"` / `"onuragtas/reqost"` and are overridable at build time via `-ldflags "-X reqost/internal/update.Version=… -X reqost/internal/update.RepoSlug=…"`. The CI workflow always injects them.

### Important gotchas

- The Go module is `reqost`; internal imports are `reqost/internal/...` and generated bindings live under `frontend/bindings/reqost/`.
- Backend service methods must stay JSON-serializable and binding-friendly; after any signature change, run `task bindings` or the frontend will fail to resolve the import.
- A large committed `collection-50k.json` (~15MB) exists as test data.
